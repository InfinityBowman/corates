0. The invoice status bug — confirmed, and here's the exact scenario

handleInvoicePaymentSucceeded (packages/stripe-purchases/src/routes/handlers/invoiceHandlers.ts:78-85) looks up the row by subscription ID and unconditionally sets status: 'active' — it never checks existing.status. Meanwhile cancellation state is written elsewhere (Better Auth's webhook path, and syncStripeSubscription.ts which correctly derives status from a fresh Stripe read).

Stripe does not guarantee event ordering, and it retries failed webhook deliveries for up to ~3 days. So: user pays their final invoice, then cancels immediately (or the events simply arrive out of order) → the DB row is set to canceled → a late-delivered or retried invoice.payment_succeeded for that same subscription flips it back to active. Because a dead subscription generates no further events, nothing ever corrects it — the org keeps paid features indefinitely until something happens to call the sync path. The fix is small: either skip the update when existing.status === 'canceled' (or endedAt is set), or do what syncStripeSubscription does and write status from a canonical Stripe re-fetch instead of assuming the event implies "active."
